# dl-xhs-benchmark 设计文档（原 dl-xhs-distill 重构）

状态：待用户审阅

## 1. 背景与定位转变

`dl-xhs-distill` 原本的定位是"蒸馏小红书博主的整个账号"，产出 IP 定位（账号定位三问/差异化人设/商业模式）+ 运营策略（更新频率/发布时段/账号阶段）+ 内容形式三层框架的分析报告和创作 Skill。

这个定位判断错了。这个 skill 的实际使用者是**小红书线索商家**（和 `dl-xhs-winback` 服务同一批人），他们的真实需求是：**做投放笔记时找对标内容参考，二次创作**。不需要理解一个博主的账号是怎么运营起来的，只需要吃透"对标内容怎么写的"，然后套自己的业务去二创。

因此整体重构为：

- **改名**：`dl-xhs-distill` → `dl-xhs-benchmark`
- **去掉账号层分析**：IP 定位、运营策略两层整体删除，不留简化版、不留背景参考
- **只保留笔记结构层分析**，并把它拆解得比原来的"内容形式"层更细致
- **两种采集入口**，服务两种真实使用场景

## 2. 两种模式

### 模式一：对标博主

用户给一个博主主页链接/分享短链。AI 采集这个博主的笔记（默认按点赞数取前 20 篇，用户可自定义数量），分析这批笔记的写作规律。

### 模式二：指定笔记

用户直接发送笔记链接，一行一条。这些笔记**允许来自不同博主**——线索商家收藏的对标笔记本来就可能来自多个不同账号，只是套路类似。AI 逐条采集这些指定笔记，用同一套下游分析逻辑处理。

### 入口交互

不再像旧版那样让 AI 自动判断走哪个模式。SKILL.md 一开始先问用户选模式（"分析一个对标博主的账号" / "分析你指定的几篇笔记"），选完模式再要求对应的输入：

- 模式一：博主链接 + 采集篇数（默认高赞前 20，可覆盖）
- 模式二：提示用户逐行发送笔记链接

## 3. 架构：单一 skill，两个入口，共用下游管线

评估过三种架构（单 skill 双入口共用管线 / 拆成两个独立 skill / 全自动单命令弱化 phase 可见性），采用**单 skill 双入口共用下游管线**：两种模式只在"采集"这一步分叉，产出同样字段结构的 `notes_details.json` 之后，后续的分析、AI 任务生成、报告/Skill 生成、质量校验全部共用同一套代码，不关心笔记是哪个模式采来的。

理由：
- 改动集中在采集层，下游逻辑改一次两个模式都受益
- 保留现有的断点续采（checkpoint）能力和"生成前先跑质量校验"的可干预节点，这是已经验证过的能力，全自动单命令会丢掉这些
- 避免报告生成/质量校验逻辑维护两份（拆成两个 skill 的问题）

### 采集层改动

- `scan_blogger.py`：去掉原本的"快速(1/3)/推荐(1/2)/深度(全量)"三档划分，改成扫描博主全部笔记的轻量列表（含点赞数等基础指标），按点赞数排序算出前 N 篇（N 默认 20，用户可指定覆盖），把这 N 篇的 note_id 交给下一步
- `crawl_blogger.py`：只负责按 `scan_blogger.py` 选出的 note_id 列表拉取笔记详情，逻辑不变，保留 checkpoint 断点续采
- 新增 `crawl_notes.py`：输入是用户发的笔记链接列表，逐条解析出 note_id，单独调用 TikHub 详情接口拉取（不需要 scan 这一步，因为没有"总篇数"概念，用户给几条采几条）；产出 JSON 字段结构与 `crawl_blogger.py` 保持一致
- 模式二的输出文件不再以 `<user_id>` 为前缀命名（可能有多个不同 user_id），改用自动生成的批次标识（日期 + 篇数，如 `20260715_5notes`），不用额外询问用户

## 4. 分析层：单篇标注 → 框架分组 → 提炼规律 → 可执行规则

参考了用户提供的一份"多篇文章风格分析"提示词，其方法论（多篇交叉验证 → 提炼跨文本规律 → 构建可执行的风格模型）值得借鉴，但其维度设计（思维模式/文化底蕴/韵律节奏等 14 个宏微观维度）是给长文写作模仿设计的，不适合小红书短平快投放笔记。因此只吸收方法论，不引入维度，也不额外增加"说服逻辑/情绪基调/读者关系"这类宏观层——这些内容与已有的"框架"（本身就是论证结构）和"CTA"（本身就是读者关系触发点）重复，属于为了显得严谨而堆维度。

### 分析维度（6 项，替代旧版"内容形式"层）

标题公式 / 开头模式 / 中间结构 / 结尾方式 / CTA 类型 / 叙事框架

### 分析流程

1. **单篇标注**：analyze.py 保留现有均赞/均藏/均评/藏赞比/图文视频比/标签频率/标题公式分类（沿用现有 5 类正则规则）；去掉发布节奏统计（账号层指标，不再需要）。全部 N 篇笔记的完整文案（不再只是 TOP10 摘 200 字）都交给 AI，因为每一篇都要被归类到某个叙事框架
2. **框架分组**：AI 读完全部笔记全文后，把每篇笔记归类到一个叙事框架（痛点型/测评型/教程型/……，具体分几类由 AI 读完这批笔记后归纳，不是写死的分类表）
3. **提炼规律**：不区分"是因为作者不同还是选题不同造成的差异"——只要在这个框架分组里反复出现、值得抄的写法规律，就直接提炼成可执行规则；框架组内笔记数 ≥ 2 时做交叉提炼，只有 1 篇时直接把这篇的拆解结果当结论用（跳过对比步骤，避免"和自己对比"这种无意义操作）
4. **代表作精读**：每个框架挑 1-2 篇高赞代表作做逐句/逐段拆解（替代旧版"TOP10 跨笔记精读"）
5. **产出可执行规则**：结果导向，直接落成"标题公式怎么写/开头怎么起/中间怎么推进/怎么收尾/CTA怎么写"这类可直接套用的规则，不停留在"观察到的现象"描述

### 不采集用户业务信息

报告和 Skill 只产出对标笔记的套路规律（标题公式/开头模式/CTA类型等），不主动询问用户自己的产品/卖点/目标人群。用户以后每次要写新笔记，自己在对话里说明业务，由 Skill 现场套用规律，不依赖蒸馏时预先问过的业务信息。

## 5. 产出物

### 命名与文案

统一把"蒸馏"相关措辞换成"拆解"：

| 旧文件名 | 新文件名 |
|---|---|
| `<博主展示名>_数据底稿.md` | 不变 |
| `<博主展示名>_AI蒸馏任务.md` | `<博主展示名>_AI拆解任务.md` |
| `<博主展示名>_蒸馏报告.html` | `<博主展示名>_拆解报告.html` |
| `<博主展示名>_创作指南.skill/` | `<博主展示名>_写作指南.skill/` |

模式二（多篇不同博主笔记）没有单一博主展示名可用，文件名标识自动生成（日期 + 篇数，如 `20260715_5篇分析_拆解报告.html`），不询问用户。

### HTML 报告视觉规范

配色/字体/圆角卡片这套"蒸馏简报"视觉风格本身保留不变（近白底 + 靠紫强调色 + 无衬线字体 + 大圆角浮起卡片），只改：

- 风格名字叫法从"蒸馏简报"改为呼应新定位的说法
- 招牌的侧边进度轨结构：原来跟随 IP定位/运营策略/内容形式三层滚动高亮，现在改成**框架分类为一级节点，标题/开头/中间/结尾/CTA 为二级节点**，外加一个"代表作逐句拆解"节点

### 创作 Skill 文件夹

内容章节从"IP定位+运营策略+内容形式"三层可执行结论，改成围绕 6 个笔记结构维度的可执行规则（按框架分组展开）。融合风格模式下的可选混合维度清单更新为：标题公式 / 开头模板 / 中间结构 / 结尾方式 / CTA 策略 / 框架选择 / 语言习惯（去掉旧版"发布节奏建议"这一项，因为不再分析发布节奏）。其中"语言习惯"不是第 4 节之外新增的分析步骤，而是从标题/开头/中间/结尾/CTA 各项分析过程中观察到的用词/句式偏好汇总而成，和旧版清单里"语言习惯"独立于三层框架之外列出的做法保持一致。

其余不变：仍需支持"1:1 模仿"和"融合风格（保留用户自己部分维度）"两种调用方式，文档开头需说明两种用法。

## 6. 质量校验更新

`quality.py`：

- HTML 报告：不再检查"IP定位/运营策略/内容形式"三个关键词锚点，改成检查报告正文能找到至少 1 个具体框架名称 + 6 个维度关键词（标题/开头/中间/结尾/CTA/框架）均有实质内容
- Skill 文件夹：是文件夹、`SKILL.md` 非空、两种调用方式说明齐全这些检查项不变；融合维度清单校验从"至少 7 个"改成对应新的 7 项维度列表

## 7. 改名影响范围

纯改名/文案替换，不涉及新逻辑：

- 目录 `skills/dl-xhs-distill/` → `skills/dl-xhs-benchmark/`
- `SKILL.md` 的 `name` 字段、description、触发词（去掉"蒸馏博主""拆解博主账号"这类措辞，换成"拆解对标笔记""对标笔记二创"）
- `.claude-plugin/marketplace.json` 里的 plugin 条目
- 根目录 `README.md` 的工具箱表格和安装命令
- `references/产出物质量标杆.md` 里的验收标准文案
- 所有代码/文档里"蒸馏"→"拆解"（文件名、变量名、注释、日志输出提示文字）
- 本地 Token 配置路径 `~/.dl-xhs-distill/tikhub_config.json` → `~/.dl-xhs-benchmark/tikhub_config.json`，**不做迁移**，改名后首次运行会重新引导注册一次 Token（几十秒的事，注册免费，没必要为了省一次引导专门写迁移代码）
- 旧 `SKILL.md` 里有一处指向 `docs/2026-07-14-dl-xhs-distill-design.md` 的引用，该文件在仓库里实际不存在（悬空引用）。新 `SKILL.md` 改写时顺带把这处引用指向本设计文档，或直接去掉这句引用

## 8. 测试改动

- `tests/test_analyze.py`：跟随新的统计字段调整（去掉发布节奏相关用例，标题公式分类用例保留）
- `tests/test_quality.py`：跟随新的校验关键词/维度清单调整
- `tests/test_common.py`、`tests/test_tikhub_client.py`：基本不受影响，采集层的通用工具函数（HTTP 客户端、限速、checkpoint 读写）没有变化

## 9. 明确不做的事

- 不做账号层分析（IP定位/运营策略），无论简化与否
- 不主动采集/询问用户的业务信息
- 不引入长文风格分析提示词里的宏微观 14 维度体系，也不额外加"说服逻辑/情绪基调"这类宏观层
- 不对旧版 Token 配置路径做迁移
- 模式二不要求笔记来自同一博主，也不因此单独设计"跨博主一致性"分析逻辑——统一按"这批笔记的共性规律"处理

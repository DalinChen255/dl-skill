# 浏览器页面约定

以下结构来自 2026-06-21 的已登录小红书专业号私信通页面。每次运行仍须先以最新 DOM 快照验证，不要盲目复用选择器。

## UID 搜索与会话列表

- 页面 URL 路径：`/im/multiCustomerService`
- 可见列表：高度大于 0 的 `.vue-recycle-scroller.scroller`
- 会话：`.sx-contact-item[data-key]`
- 会话键：可能是 `Total-{user_id}` 或 `Total-{user_id}-{account_id}`。页面也可能出现 `Active-{user_id}`、`Favorite-{user_id}` 等前缀；业务身份仍由 `user_id + account_id` 决定。
- 当 `Total-` 键只有 user_id 时，必须打开会话后从当前会话 `jarvis-msg-` ID 推断 account_id，并用右侧“小红书uid”确认 user_id；未确认 account_id 不得发送。
- 表格跟进的主路径是 UID 搜索复核。UID 来自 `线索管理 > 用户列表` CSV，且必须已经按本轮确认账号范围过滤 `归属账号`；搜索只是“找人”，不负责“确认可发”。
- UID 搜索打开会话后，必须核对右侧“小红书uid”、实际账号 ID、留资、回复方向、负面、同行/unknown 和时间条件；不能因为 CSV 显示无留资就直接发。
- 虚拟列表只渲染可见区域附近的项目。滚动后 DOM 节点会被复用，禁止按位置或节点对象长期保存身份。
- 判断列表项是否当前可见时，必须用“会话项矩形 ∩ 列表容器矩形 ∩ 浏览器视口”的交叉面积；禁止只用 `display != none`、`visibility != hidden`、宽高或 Playwright 可见性判断，因为屏幕外残留节点可能仍有尺寸。
- 新消息会把会话移到顶部。使用已见键集合，不以列表顺序作为进度。恢复中断扫描时，可从状态库已写入的候选 key 初始化已见集合，但仍必须继续滚动到底。
- 列表滚动扫描只作为内部兜底使用。扫描兜底阶段必须直接点击当前可见的 `Total-` 会话项并向下滚动遍历，禁止在扫描建表阶段用搜索框跳 UID 或昵称。
- 本月扫描兜底只负责本月/近期私信窗口；历史覆盖不再靠长期滚动发现会话。历史覆盖使用最新导入的用户列表 UID 池，再按 UID 搜索逐个复核。
- 队列发送阶段默认优先使用队列里已保存的 `user_id` 搜索。这个 `user_id` 可能来自 CSV UID 复核，也可能来自内部扫描兜底读取到的右侧“小红书uid”。UID 搜索不可用或不稳定时，才清空搜索框并把全量会话列表回到顶部，再向下遍历当前真实可见会话项，按队列 `conversation_key` 匹配。
- 使用队列保存的 `user_id` 搜索兜底时，输入 UID 后如果出现搜索结果小框/浮层 `.search-contact-item`，不要先按 Enter，也不要先滚动或遍历左侧真实会话列表；必须优先点击浮层里的可见结果打开会话，再用右侧“小红书uid”和当前消息 `account_id` 复核。搜索结果小框有时不挂在 `.user-list` 下面；如果 legacy 容器和浮层容器同时存在，必须按真实可见面积读取浮层结果。
- 如果同一个 UID/昵称出现多个浮层结果，先点浮层第 1 个并复核；若 UID 不对或实际账号不在本轮授权账号范围内，先点击消息头/空白区域让搜索框退出当前结果状态，再回到搜索框全选、删除、逐字重新输入同一 UID 展开浮层，然后点第 2 个，以此类推，遍历当前可见浮层结果。不要只用程序填值覆盖输入框；那可能不会重新弹出浮层。单个 UID 最多尝试 20 个浮层结果以防死循环。不能只检查全量会话列表的可见项后就判定“搜索无结果”。
- 浮层可能出现 5 个以上同 UID 结果，且其中部分结果需要滚动浮层容器才会出现。读取浮层结果时必须收集可见项和滚动后出现的项；点击第 N 个结果前应恢复到该结果对应的浮层滚动位置，并按滚动后重新读取到的候选中心坐标点击。不要只用 `.nth(N)` 或旧 DOM 节点点击；浮层/虚拟列表可能复用节点，导致第 4/5/6 个候选实际又点回上方 3 个。不得因为初始视口只看到 3 个结果就停止。
- 只有浮层不存在、浮层全部点完仍不匹配，或浮层点击不可用时，才读取左侧真实会话列表里的 `.sx-contact-item[data-key]` 兜底。如果搜索后的真实列表已经出现队列保存的 `user_id + account_id` 精确项，或同 UID 的白名单账号项，应点击该列表项；列表里可见的账号标签（如 账号A/账号B/账号C）只用于辅助排序，最终身份仍以 `data-key`、右侧“小红书uid”和当前消息 `account_id` 复核为准。
- 禁止在发送阶段读取当前页面右侧正在显示的“小红书uid”并拿它去搜索。找不到目标客户时，当前右侧面板可能仍停留在上一位或其他客户；该值只能用于打开候选会话后的身份复核，不能作为搜索输入来源。
- 会话键形态从 `Total-{user_id}` 变为 `Total-{user_id}-{account_id}` 或反向变化时，只有队列中该 user_id 唯一且账号 ID 不冲突才可点击；点击后仍须核对右侧 UID 和消息 account_id。
- 搜索结果只负责“找人”，不负责“确认可发”。点击搜索结果后必须以右侧“小红书uid”和当前消息 account_id 为准；UID 必须一致，account_id 必须在本轮白名单内。
- 历史 UID 来自线索管理 CSV 时，队列项一开始可能没有 account_id。搜索打开任一结果后，只要右侧 UID 正确且当前消息 ID 证明实际 `account_id` 属于本轮白名单，就可继续复核；非白名单账号必须继续找下一个结果或记为技术失败，不能因为搜索结果出现该 UID 就扩大本轮账号范围。
- 消息区可能短暂残留其他账号的同 UID 消息。队列保存的 `account_id` 只是候选来源，不是发送硬限制；实际账号可以是本轮授权账号范围内的任一账号。为了避免旧残留误导，允许优先按队列账号精确核验；精确核验失败后，仍可放宽到任一本轮白名单账号。不得只按消息区出现次数最多的 account_id 推断实际账号。
- 浮层可见结果未确认时，不得立即放弃；如果搜索后的可见会话列表已经出现同 UID 或同 `user_id + account_id` 的真实会话项，必须继续点击列表项复核，仍然以右侧 UID 和消息 account_id 为准。
- 列表兜底时，队列保存的 `user_id + account_id` 精确项优先级最高；其他白名单账号的同 UID 会话其次；非白名单账号不得作为可发送定位结果。
- 技术定位失败只记录可重试事件，队列项保持 pending。不得把“当前不可见”“搜索无结果”当成客户不符合发送条件。
- 同一个 UID 可能在多个账号会话中出现；UID 搜索打开任一结果后，如果右侧 UID 正确且消息 ID 证明实际 `account_id` 属于本轮白名单，应继续按当前会话重新验证留资、回复、负面、can-send 和发送控件。符合条件则发送，并把发送记录写为实际 `account_id`；不再因为实际账号不同于队列来源账号而卡住。未授权账号（例如本轮未确认的账号）即使 UID 正确也不能发，应继续点下一个浮层结果。
- 真实会话列表滚动时，如果页面里有多个 `.vue-recycle-scroller.scroller`，优先选 item 数更多、scrollHeight 更大的那一个，并尽量用 `scrollTo()` 直接驱动真实列表，避免鼠标滚轮落到右侧消息区或搜索浮层。
- 日期定位模式可以大步滚动，但只能用于内部扫描兜底或人工定位；不得把被大步跳过的会话视为已扫描。正式扫描必须使用严格模式处理可见会话并按完成条件收尾。
- 不要用日期定位或滚到底来证明表格/历史覆盖完成。覆盖完成的依据是状态脚本里的 UID 复核窗口完成标记。
- 执行器进度字段 `scanProgress.visibleOldestDate` 来自当前可见会话文本中的日期标签，如 `6月25日`、`05-31`、`昨天`、`刚刚`。它用于操作进度提示，不用于客户资格判断。

## 消息区

- 当前标题容器可见类名：`.message-header`
- 消息容器 ID 前缀：`jarvis-msg-`
- 当前观察到两种 ID 形态：`jarvis-msg-{user_id}.{account_id}.{message_id}` 和 `jarvis-msg-{account_id}.{user_id}.{message_id}`。四个授权账号并不完全一致，读取时必须同时兼容两种顺序。
- 客户消息：消息容器类名 `left`
- 我方消息：消息容器类名 `right`
- 系统消息：消息容器类名 `center`
- 时间：`data-timestamp`
- 类型：`data-msg-type`
- 历史加载入口：可见文本“加载更多”

Vue 虚拟复用可能让其他会话的消息短暂残留在同一个消息滚动区域。读取消息时必须同时匹配当前 user_id 和 account_id，并兼容两种排列顺序；不能抓取页面中所有 `jarvis-msg-` 后直接分类。

读取右侧客资区的掩码手机号、微信号或留资标签之前，必须先确认右侧显示的“小红书uid”与当前会话 user_id 一致，并且当前消息 ID 能证明 `account_id`。未完成身份核对时，全页面输入框和文本都可能属于上一客户，不能作为留资依据。

## 发送区

- 输入框的可访问名称：`按 "回车键" 发送信息...（输入/唤起快捷回复）`
- 按钮名称：`发送`
- 空输入时按钮禁用。

填写和点击之前都要使用最新快照，确认定位器恰好匹配一个元素。填入文案后必须读取输入框 value，确认与原文案完全一致后才能点击发送。发送后以新增的右侧消息气泡、文案一致、新消息 ID 和新时间戳共同验证。页面若自动插入空白或换行，可在压缩空白后一致时判定文案一致；不得放宽“新消息 ID + 新时间戳 + 当前 user_id/account_id”三项身份校验。

小红书有时会先生成 `jarvis-msg-{13位时间戳}-{随机数}` 这类临时消息 ID，不带 `user_id/account_id`。临时 ID 只允许用于发送后成功验证：发送前必须已确认右侧 UID 和实际账号，发送后右侧 UID/账号仍一致，且该临时 ID 是发送前不存在的新右侧消息、时间戳晚于发送前、文案完全一致。临时 ID 不得用于扫描分类、候选身份推断或账号归属推断。

## 页面现有信号

- “留客资”是强留资信号。
- “访客联系方式已自动记录至「聚光平台-线索管理」”是强留资信号。
- “对方已点击你的企业微信联系卡”按已留资处理。
- “对方已提交信息”按已留资处理。
- 当前客资区出现掩码手机号或掩码微信号时按已留资处理。
- 已发送留资卡或企业微信名片，不等于客户已经留资。
- “[超时未回复]”不能用于判断客户未回复我方。
- “对方关注或回复你之前，24小时内最多只能发1条文字消息”是平台限制提示；最终仍以是否出现新我方消息和页面错误状态判断该次发送结果。

## 内部扫描兜底终止条件

只有内部扫描兜底才使用这些条件。表格跟进不需要滚动列表到底；它以 UID 复核窗口完成作为覆盖依据。

只有满足以下条件才认为会话列表遍历完成：

1. 已到可见滚动容器底部。
2. 正式扫描连续五次稳定检查没有发现新业务会话键；只读探针或发送列表兜底可以使用更少稳定次数。
3. 两次扫描之间滚动高度没有增长。
4. 没有正在进行的其他批量发送导致列表持续重排。

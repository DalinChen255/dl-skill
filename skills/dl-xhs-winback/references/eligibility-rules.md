# 筛选与发送规则

## 状态枚举

### lead_status

- `has_lead`：出现以下任一信号：`留客资`、`对方已点击你的企业微信联系卡`、`对方已提交信息`、联系方式自动记录、当前客资区有掩码手机号/微信号，或客户明确发送可用手机号/微信号。
- `no_lead`：完整历史中没有任何留资强信号。
- `unknown`：历史未加载完整、客户发送过无法读取的图片/卡片/文件且其中可能包含联系方式，或页面状态不明确。用户人工确认图片上有联系方式时，改判 `has_lead`。

判断客户是否留资时，只使用客户左侧消息、已确认同一 user_id 的右侧客资区、系统强信号。不要把我方右侧已发送文案里的手机号、微信号、企微卡或引导语当成客户留资。

线索管理导出的用户列表 CSV 是默认发现来源，同时只作为保护信号和 UID 来源：

- CSV 必须先按本轮确认账号范围过滤 `归属账号`；未确认账号的行不得进入 UID 复核池。
- 在本轮账号范围内，`手机号`、`微信号`、`最近留资时间`、`留资方式` 任一有值，按 `has_lead` 保护，后续候选和发送前验证都必须排除该 UID。
- CSV 中没有留资字段，只代表“值得进入 UID 复核池”，不等于 `no_lead` 已经可发送。
- `1A 最新单表` 只允许使用最新 import id + 本轮账号范围交集对应的无留资 UID，避免旧表格、旧复核结果或非授权账号混进当天任务。
- `1B 历史多表` 只在本轮账号范围内合并多张表；同 UID 在任一授权账号行已留资就保护，同 UID 只在未授权账号留资不直接污染本轮目标池。
- UID 必须在私信页按 UID 搜索打开后重新确认右侧 UID、实际账号 ID、当前留资信号、消息历史、回复方向、负面和同行/unknown。只有浏览器复核写入候选后，才允许进入队列。

### party_status

- `customer`：没有明确同行、商业服务号或向我方推销的证据。无需额外出现工厂、产品、课程、价格等购买意向关键词；简单咨询、打招呼也按非同行处理。
- `peer`：客户左侧真实消息明确表明对方在推广 SaaS、代运营、广告合作、接广、代理、培训、同行服务或其他面向我方销售的业务。
- `unknown`：历史无法读取、当前会话身份无法确认、消息内容损坏，或昵称/标题/右侧资料/客户消息出现商业服务号、疑似同行、行业内容号信号。`unknown` 不进入自动发送队列，先人工复核。

只根据昵称不能把人直接判成 `peer`；但昵称、标题或右侧资料出现疑似商业服务信号时，必须判为 `unknown`，不得按普通客户自动发送。自动召回不是法庭判案，不需要证明 100% 是同行才拦截。

判断同行/服务商/推销方时，只使用客户左侧真实消息、客户昵称/标题、已确认同一 user_id 的右侧资料。不要因为我方右侧文案里出现“代运营”“服务商”“获客”等词，把客户判为同行。

商业服务号/疑似同行信号包括但不限于：AI 工具/AIGC/创作工具、海外落地、企业出海服务、出海拓客、获客、询盘、线索、资源对接、项目对接、代运营、投放、广告、SaaS、软件系统、建站、独立站、培训、课程、陪跑、招商、代理、服务商、行业内容号表达如“谈外贸/外贸之路/跨境老鹰/跨境观察/出海日记/看世界”。命中这些信号时默认 `party_status=unknown`，需要人工确认后才能自动触达。

### reply_status

排除 `center` 系统消息、撤回提示、卡片和进入私信提示后，按时间寻找最后一条真实对话：

- `awaiting_customer`：最后一条真实消息来自我方，之后没有客户回复。
- `awaiting_us`：最后一条真实消息来自客户。
- `unknown`：没有真实消息或发送方向无法确定。

### negative_status

- `blocked`：客户明确拒绝、要求不要联系、投诉、辱骂、表示被骗或出现明显负面反馈。
- `safe`：没有负面或拒绝信号。
- `unknown`：语义不清。

负面状态优先级最高；`blocked` 永久排除自动召回。

判断拒绝、投诉、辱骂、停止联系等负面状态时，只使用客户左侧消息。不要因为我方文案里出现“不需要可以不用回复”等表达，把客户判为拒绝。

## 资格真值表

只有以下组合允许进入候选：

| lead | party | reply | negative | 结果 |
|---|---|---|---|---|
| no_lead | customer | awaiting_customer | safe | 继续检查时间和去重 |
| 其他任意组合 | 任意 | 任意 | 任意 | 排除 |

时间和去重还必须满足：

- 会话所属 `account_id` 必须属于本轮确认的账号 ID 白名单。
- 最后一条真实消息距当前至少 24 小时。
- 同一活动内，同一 user_id 只能成功发送一次，即使出现在多个账号。
- 任意活动的最近成功发送距当前至少 24 小时。
- 回复状态按当前 `user_id + account_id` 会话判断；同一 user_id 在其他账号里出现客户回复，不自动排除当前账号会话。
- 已留资、同行/服务商/推销方、疑似同行/商业服务号、拒绝/投诉/辱骂/停止联系等强信号按 user_id 做保护性排除；同一 user_id 在任一白名单账号出现这些强信号时，自动召回应排除该用户，除非本轮人工复核明确覆盖。

## CSV/UID 复核后重新验证

CSV 导入和 UID 复核得到的候选只代表当时状态；进入发送队列后，每个客户发送前都必须重新读取当前会话并重新计算状态。

表格跟进下，必须先导入用户列表 CSV，再完成本轮账号范围过滤后的 UID 逐个复核并写入窗口完成标记，才能进入发送队列。`1A 最新单表` 使用 `user_list` scope，要求最新 import id 足够新；`1B 历史多表` 使用 `history` scope，不要求最新 cutoff。CSV 导入不是复核完成，也不是发送许可。

内部 `month` 扫描兜底下，必须先完成本月私信窗口覆盖，才能进入发送队列。只读探针不算正式扫描；扫描 chunk 返回 `chunk_limit` 或超时只表示暂停，不表示完成。允许两种完成依据：正式扫描返回 `done: true` 且停止原因是 `completed`，或严格扫描已覆盖本月边界并写入 `mark-current-month-window`。

`history` scope 用于 `1B 历史多表` 和内部旧历史兼容；都必须完成历史 UID 复核窗口。内部 `combined` 兼容 scope 还必须满足本月窗口覆盖，不作为用户主模式展示。

旧每日增量扫描是内部优化例外：只有状态脚本确认 48 小时内同账号范围的全量基线、并完成 `incremental-plan -> seed-incremental-candidates -> 窗口严格扫描 -> mark-incremental-window` 后，才可显式传 `allowIncompleteScan:true` 建队列和发送；它不作为用户主模式展示，不能绕过 UID 复核。

发送前重新读取当前会话并重新计算状态。出现以下任一情况就跳过：

- 客户刚刚回复。
- 新增留资记录。
- 新增拒绝或负面消息。
- 右侧 UID 与候选 `user_id` 不一致，或实际打开会话的 `account_id` 不在本轮白名单内。实际账号与队列来源账号不同不单独构成排除理由。
- 输入框或发送按钮不可用。
- 已被其他活动或账号触达。
- 该 user_id 存在未人工解除的 `uncertain` 发送记录。同一 batch 已出现 `uncertain` 时，必须先人工处理该不确定项；状态脚本会阻塞继续读取 pending 队列和发送，避免可能已发送但未验证的批次继续扩大风险。
- 历史导入账本显示该 user_id 在本轮账号范围内已留资。

## 快速排除顺序

1. 表格导入时，先按本轮账号范围过滤 `归属账号`，再用 CSV 留资字段保护已留资 UID。
2. UID 复核或内部扫描打开会话后，先检查当前消息界面和客资区；命中任一留资强信号就跳过，不加载更多。
3. 读取客资区前先确认右侧“小红书uid”等于当前会话 user_id，避免把上一会话的掩码联系方式误判给当前客户。
4. 只有快速检查未命中时，才加载历史并判断同行/疑似同行、最后回复方向、负面状态和 24 小时条件。昵称/标题/右侧资料命中商业服务号信号时，直接记为 `party_status=unknown`，不要进入自动发送队列。

## 文案与账号授权

用户提交本轮完整文案后，还必须确认账号 ID 范围。只有「完整文案 + 账号 ID 范围」都确认后，才允许生成无副作用 `operator-plan`；真正导入表格、启动活动、复核 UID、建队列或发送，还必须等用户明确选择 1/2/3/4，并在选择 1 时明确 `1A/1B`。不要再要求候选人数确认或批次确认。

文案变化分两种：

- 默认每日新任务：文案变化时先跑 `operator-plan`，展示推荐模式并等待用户选择；用户选择 `表格跟进` 后还必须确认 `1A 最新单表` 或 `1B 历史多表`，之后才导入 CSV 并建立或复用对应 scope 活动；选择 `继续未完成任务` 时先按状态恢复旧活动；选择 `处理异常/补发` 或 `只查状态` 时不得为了新文案启动新活动。
- 用户明确要求沿用当前已扫描候选池/当前队列继续发送：必须先执行 `update-message` 更新状态账本中的当前活动文案，并由发送执行器逐条读取状态账本文案；不得只改浏览器 session 变量。

同文案去重必须使用每条发送记录保存的 `message_hash_snapshot`，不能用 campaign 当前 `message_hash` 反推历史发送版本。
新的 `sent`、`failed`、`uncertain` 发送尝试必须保存实际发送文案快照；历史记录缺失快照时只能在看板和报告里标记为缺失，不得把缺失快照的历史发送冒充成当前文案发送。
